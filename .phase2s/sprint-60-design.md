## Problem Statement

Phase2S has 29 built-in skills. There is a significant quality gap between the best
and the worst. The strong skills (adversarial, satori, slop-clean, debug, deep-specify)
share a common structure: clear protocol, defined output format, a verify step,
and saved artifacts. The weak ones give instructions but no structure — users can't
rely on consistent output, can't pick up where they left off, and can't catch when
the skill made things worse.

Separately: 28 of 29 skills have no typed `inputs:` field. Skills that take arguments
(like `/freeze <dir>`, `/debug <file>`, `/review <scope>`, `/slop-clean <path>`)
have to infer from conversation context instead of declaring what they accept. This
matters most for MCP tool descriptions — right now all 28 look identical to Claude Code
(just the one-liner description, no parameter info).

## What Makes This Important

Every skill invocation is a user trusting the system to do the right thing. When
`/ship` doesn't run tests before committing, or `/review` makes inline fixes that
break the build without checking, or `/docs` generates annotations with type errors —
users stop trusting skills. They go back to doing it manually.

The typed inputs gap means users of Phase2S via MCP have no way to know what
arguments a skill accepts without reading the SKILL.md directly.

## Save Path Convention (all skills that write artifacts)

All skills that write to `.phase2s/` must instruct Claude to:
1. Create the directory first: `mkdir -p .phase2s/<skillname>/`
2. Write the file: `<YYYY-MM-DD-HHMM>-<slug>.md` where slug is derived from branch name or topic
3. Tell the user the full path after saving

This prevents silent failure when the directory doesn't exist yet.

## Constraints

- Skill files are `.phase2s/skills/*/SKILL.md` — pure markdown, no compilation
- C-rated skills are intentionally thin. Do not over-engineer them. `unfreeze` at
  21 lines is correct.
- Changing skill instructions changes skill behavior. Each rewrite must be tested
  manually by invoking the skill after changes.
- `plan` is the typed inputs reference. Its inputs: field shows the format to follow.
- 1522 vitest tests must remain green. Skill file changes don't affect the test suite
  unless runtime code (skill loading, MCP tool generation) is also touched.

## Approaches Considered

### Approach A: Inputs-only sprint

Effort: S. Risk: Low.
Pros: MCP tool descriptions improve immediately across all skills.
Cons: Doesn't fix the real quality gaps — review still doesn't verify, ship still
doesn't test.

### Approach B: Quality-first targeted rewrite (CHOSEN)
Audit all 29 (done above). Fix the 6 D-rated skills. Add typed inputs as part
of each rewrite, then add inputs to remaining skills.
Effort: M. Risk: Low-Med.
Pros: Fixes actual problems. Produces audit matrix as permanent artifact.

### Approach C: Quality rewrites + A1 sibling cancellation
Same as B plus AbortController fast-pause in parallel-executor.ts.
Effort: M-L. Risk: Med.
Deferred: two independent tracks, added complexity.

## Recommended Approach

**Approach B — Quality-first targeted rewrite.**

Execution order:
1. Commit the audit matrix doc (this file) on sprint-60 branch
2. Fix `review` first — highest daily-use skill with the biggest gap:
   - Add: verify step — run `npm test` unconditionally at the end of every review,
     whether or not inline fixes were applied. Serves as both a safety gate after
     fixes and a baseline health check for observation-only reviews.
   - Add: save output to `.phase2s/review/<YYYY-MM-DD-HHMM>-<branch-slug>.md`
     (consistent with how debug and health save — date + time prefix to avoid
     same-day collision, branch as slug)
   - Add: typed inputs (scope: optional path/glob)
   - Add: `{{scope}}` used in body where scope is provided (skip section if absent)
3. Fix `ship`:
   - Add: run the test command before commit. Detection order: check `package.json`
     scripts for a `test` entry and run `npm test`; if no test script exists, skip
     with a warning ("No test script found — skipping test gate"). Block on failure
     with message: "Tests failed — fix before shipping." No interactive prompt;
     hard block is the right behavior (consistent with slop-clean's baseline check).
   - Add: typed inputs (none needed — operates on git status)
4. Fix `docs`:
   - Add: run `tsc --noEmit` after generating annotations. Report type errors.
     Note: only applies to TypeScript projects. If no `tsconfig.json` is present,
     skip with a note ("Not a TypeScript project — skipping type check").
   - Add: save generated docs summary to `.phase2s/docs/<YYYY-MM-DD-HHMM>-<slug>.md`
   - Add: typed inputs (path: required), `{{path}}` used in body
5. Fix `investigate`:
   - Add: save investigation log to `.phase2s/debug/<YYYY-MM-DD-HHMM>-investigate-<slug>.md`
     (same directory as `debug` saves — `investigate-` prefix distinguishes from
     debug session outputs; time component avoids same-day collision)
   - Add: typed inputs (bug: description string), `{{bug}}` used in body
6. Fix `tdd`:
   - Add: save test plan to `.phase2s/specs/<YYYY-MM-DD-HHMM>-<slug>.md`
   - Add: typed inputs (feature: description), `{{feature}}` used in body
7. Improve `skill` meta-skill template:
   - Ask one additional question: "Does this skill make code changes or write files?"
   - Generated template gains structural sections based on the answer:
     - `## Output` section (always) — placeholder for output format definition
     - `## Verify` section (if yes to code changes) — "Run `npm test` to confirm nothing broke"
     - `## Save` section (if yes to file writes) — ".phase2s/<skillname>/<date>-<slug>.md path"
   - Generated frontmatter includes `inputs:` block with one placeholder arg if applicable
   - This means every new skill created via `/skill` starts at B-quality minimum
8. Add typed inputs to all remaining skills with named arguments (those not already touched)
   Use the audit matrix "Action" column as the source of truth.
   **IMPORTANT:** The runtime (`loader.ts:213`) discards empty inputs dicts — `inputs: {}`
   is functionally identical to absent `inputs:`. Do NOT add `inputs: {}` to no-arg skills.
   Skills with "add inputs: none" in the matrix: leave untouched (no edit).
   Skills with named args (e.g., "add inputs: note"): add both the `inputs:` field
   AND a `{{note}}` placeholder in the relevant spot in the skill body.
   Reference `plan/SKILL.md` for the exact `inputs:` YAML format and `{{param}}` syntax.
   Skills in this batch with real args: adversarial (plan_text), audit (scope),
   autoplan (tbd), checkpoint (note), consensus-plan (tbd), debug (bug_description + file_path),
   deep-specify (feature), explain (target), freeze (directory), land-and-deploy (tbd),
   qa (url), remember (content + type), satori (task + eval_command), slop-clean (path).
   Run `phase2s skills` after each commit to confirm the skill loads without parse errors.

Each skill fix is one commit. PR has 8+ commits, all reviewable independently.

## Success Criteria

- All 6 D-rated skills have: (a) output format declared, (b) verify step where
  applicable, (c) save output to .phase2s/, (d) typed inputs + `{{param}}` body usage
- All skills with named arguments have `inputs:` field + `{{param}}` in body
  (no-arg skills are intentionally left without an inputs: field — runtime discards empty dicts)
- Audit matrix doc committed to repo (or linked from CHANGELOG)
- `docs/memory.md` "What Phase2S writes to disk" table updated with new artifact paths:
  `.phase2s/review/`, `.phase2s/docs/`
- `npm test` — 1522+ tests passing
- `phase2s skills` runs cleanly (no parse errors) after each skill commit
- Manual invocation matrix for ship and review specifically:
  - `ship`: (a) repo with passing tests → commits, (b) repo with failing tests → hard block,
    (c) repo with no test script → warns and continues
  - `review`: (a) observation-only run → still runs npm test, (b) inline fixes applied → runs npm test
- CHANGELOG v1.34.0 entry documents which skills were updated and what changed
